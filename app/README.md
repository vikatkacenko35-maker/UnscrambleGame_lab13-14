# Лабораторная работа №13-14. ViewModel и архитектура Android-приложений

## Кратко об игре
На экране показывается слово с перемешанными буквами (например, "drnoaAi"), а игрок должен ввести правильный вариант ("Android"). За каждое правильное слово начисляется 20 очков. Всего в игре 10 слов. Если не знаешь ответ — можно пропустить слово, но очки не добавятся. В конце игры показывается итоговый результат и предложение сыграть ещё раз.

Что использовалa
Kotlin — основной язык

ViewModel — чтобы хранить состояние игры

StateFlow — для реактивного обновления экрана

Material Design 3 — для внешнего вида

JUnit — для тестов

## Что умеет приложение
Показывает перемешанное слово, которое нужно отгадать

Считает количество очков и номер текущего вопроса (1 из 10)

Позволяет вводить ответ и проверять его

При правильном ответе увеличивает счёт и переходит к следующему слову

При неправильном ответе показывает красное сообщение об ошибке

Можно пропустить слово (счёт не меняется)

После всех 10 слов показывает диалог с результатом

Можно начать игру заново

## Контрольные вопросы 

### Что такое ViewModel и зачем он нужен?
ViewModel — это специальный класс из Android Jetpack, который хранит данные, связанные с интерфейсом. Без ViewModel при повороте Activity пересоздаётся, и все переменные (счёт, текущее слово и т.д.) сбрасываются. 
### В чём разница между MutableStateFlow и StateFlow?
**MutableStateFlow** — изменяемый. Используется внутри ViewModel, чтобы обновлять данные.

**StateFlow** — неизменяемый. Его отдаём наружу (в UI), чтобы никто случайно не сломал состояние.
Паттерн с _uiState (приватное изменяемое) и uiState (публичное только для чтения) называется backing property. Это защита: UI может только читать данные, но не менять их напрямую. Всё изменение идёт через методы ViewModel.

### Почему в ViewModel нельзя хранить Context или View?

Если сохранить ссылку на Activity или Context в ViewModel, то при повороте экрана Activity уничтожится, а ViewModel останется и будет держать ссылку на старую Activity

### Чем StateFlow отличается от remember и rememberSaveable?
remember — хранит значение только пока композиция жива. При повороте экрана данные теряются.

rememberSaveable — переживает поворот экрана, но сохраняет данные через Bundle (есть ограничения по размеру и типам данных).

StateFlow в ViewModel — лучший вариант для бизнес-логики. ViewModel живёт дольше, чем Activity, и идеально подходит для хранения состояния игры.
###  Что такое data class в Kotlin и зачем он используется для UI State?
data class — это класс, который автоматически генерирует методы equals(), hashCode(), toString() и самое главное — copy().

Для UI State это удобно, потому что состояние хранится в одном месте (все переменные игры: счёт, слово, ошибка и т.д.). А copy() позволяет легко создавать копию объекта с изменёнными полями.
###  Как работает collectAsState() в Compose?
collectAsState() подписывается на StateFlow и "слушает" изменения. Когда в ViewModel обновляется _uiState, StateFlow отправляет новое значение всем подписчикам, и collectAsState() вызывает перерисовку (рекомпозицию) экрана с новыми данными.
### Зачем разделять код на пакеты data и ui?
data — данные и модели (GameUiState, список слов)

ui — всё, что связано с интерфейсом (ViewModel, экраны)

Это поддерживает принцип разделения ответственности (Separation of Concerns). Код становится чище, легче искать нужные файлы, проще тестировать, и несколько разработчиков могут работать над разными частями без конфликтов.
### Что происходит с ViewModel, когда приложение полностью закрывается(process death)?

Eсли приложение убивает система из-за нехватки памяти (process death), ViewModel тоже уничтожается. Данные в этом случае теряются. Для долговременного хранения нужно использовать другие механизмы: Room (база данных), DataStore или обычные файлы. ViewModel — это только для временного хранения в рамках одного запуска приложения.
